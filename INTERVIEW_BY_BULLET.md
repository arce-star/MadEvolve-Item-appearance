## Bullet 1: 框架复现与回测引擎

### 面试官可能问

**"论文的框架和你的实现有什么关系？"**

MadEvolve 是开源框架，我直接用它做进化调度层（迭代循环、种群管理、LLM 路由）。但我从零写的是**量化领域适配层**——回测引擎、评估器、基线策略、配置系统。论文只提供了通用框架，没有任何交易相关代码。

**"回测引擎具体做了什么？"**

四个东西：
1. **Fill Model**(Appendix A.2.1): 被动限价单，买单当 `Low < limit_price` 成交，卖单当 `High > limit_price` 成交，hit_ratio=1，成交价=limit_price 无额外滑点。
2. **PnL 分解**(Appendix A.3): `PnL_adj = q_{t+1}·δm - spread_cost - 15bps_fee - impact_cost`，每笔成交都算四项。
3. **冲击模型**(Appendix A.4, Table 9): square-root propagator，永久项(α_perm=0.005) + 瞬时项(α_trans=0.010 × G(τ))，G(τ)=(300/(τ+300))^0.5。保守设计——自冲击计入 G(0)=1。
4. **Alpha Forecaster**(Appendix B.1): 固定 Ridge(α=0.5)，3 个 EMA 特征(halflife 1/5/10) → 4 个 horizon 预测 → sum → 单一 α 信号。

**"52.7 万行数据规模多大？"**

全年 2024 的每分钟 OHLCV。BTCUSDT 1-min K线，527,040 根。训练集 2022-2023 共 105 万根，测试集 2025 共 40 万根。总计约 200 万根。

**"你为什么说'严格按论文实现'？有什么证据？"**

附录 B.2 的基线策略代码我逐行对照了：11 个构造函数参数完全一致，set_target() 的五个阶段（cost-adjusted targets → risk_reduction → lag_adjustment → inventory correction → clip）一行不差。冲击模型的 6 个参数我直接从 Table 9 抄的。

### 关键数字
- 527,040 行验证数据
- 6 参数冲击模型 (V, α_perm, α_trans, τ₀, β, δ)
- 15 bps 手续费
- PnL 包含四项分解

## Bullet 2: 性能优化

### 面试官可能问

**"42 分钟到 7 分钟，具体做了哪些优化？"**

三轮优化，从 42 分钟→7 分钟：

1. **numpy 向量化**（最大收益 ~6x）：原始 `_displacement()` 用 Python for 循环遍历历史交易算 G(τ)，每年 40 万次调用 × 每次几百笔历史 = 上亿次 Python 迭代。改为 numpy 数组操作——一次创建 taus 数组、一次 mask、一次向量化乘法求和，C 层执行。

2. **G(τ) 查表 + 滑动窗口**（~3x）：幂运算 `(300/(τ+300))^0.5` 很贵。预计算 G_table[0..86400]，按秒数直接索引。同时维护 `_window_start` 指针，只扫描过去 1 天内的交易（1440 条 vs 之前的 40 万条）。

3. **预计算价格 + 跳过空策略调用**（~1.5x）：在循环前用 `numpy` 一次性算完 52.7 万个 mid/high/low，避免每行 `DataFrame.iloc[]` 的 Python 开销。alpha 接近零且仓位很小时直接跳过 `set_passive_order_data()` 调用。

**"内存 200MB→4MB 怎么做到的？"**

根因是 Python 对象的内存开销远大于原始数据：
- `pnl_components`：40 万个 dict（每个 ~300B，Python dict 元数据 + 字符串 key + float value）→ 120MB。改为 `np.zeros(527040, dtype=float64)` → 4MB。连续 C 内存，零 Python 对象开销。
- `trades`：40 万个 TradeRecord dataclass 实例（~200B/个）→ 80MB。改为 `int` 计数器（回测打分不需要存每笔明细）。plotting 用的 equity curve 保留为简单 float list。

**"为什么 42 分钟之前这么慢？"**

最初的 Python 实现没有任何优化考量。冲击模型是逐笔交易的 Python for 循环，而且初期用 30 天 lookback 窗口——40 万笔交易 × 每次扫描 8000+ 历史交易 = 超 30 亿次 Python 迭代。加上 `pd.DataFrame.iloc[]` 逐行索引的 overhead。本质上是一个 Python 原型直接上了生产数据量。

**"怎么验证优化没有引入 bug？"**

两次回测（优化前后）的 total PnL 完全一致（-$2,304,113），证明等价。因为没改计算逻辑，只改了执行效率。

**"还有进一步优化空间吗？"**

主循环 52.7 万次 Python 迭代当前约 130 秒，占总耗时 31%。用 numba.jit 编译主循环可以再提速 5-10 倍（C 级别执行），但代码复杂度会大幅增加。

### 关键数字
- 42 min → 7 min（总体 6x）
- 200MB → 4MB（98%）
- 冲击模型函数内 ~50x 向量化加速
- 40 万笔交易、52.7 万根 K 线

## Bullet 3: 进化算法与 LLM 集成

### 面试官可能问

**"MAP-Elites 是什么？为什么需要它？"**

一种质量-多样性优化算法。传统 GA 只追求最高分 → 种群会收敛到同一个解。MAP-Elites 把策略按行为特征分到三维网格（complexity=代码长度, diversity=cosine 距离, performance=分数），每格只保留最佳。这样保证种群覆盖不同的策略风格——有短代码的、有长代码的、有结构创新的。父代选择时既选高分精英，也选"格子里唯一的那个"来维持探索。

**"岛屿模型怎么工作的？"**

2 个独立子种群各自进化，每 5 代通过 ring 拓扑交换 10% 的精英。论文用 5 岛，我用 2 岛（计算资源限制）。目的是防止全局收敛——每个岛可能发现不同的局部最优，交换后能杂交出更好的解。

**"LLM 怎么参与变异的？"**

不是随机变异——LLM 看了完整的策略代码、最近的回测指标、精英参考策略后，给出语义层面的修改。两种模式：
- Differential patch（70%）：SEARCH/REPLACE 精确替换代码片段
- Holistic rewrite（30%）：完整重写整个可变函数

框架还有一个 UCB Bandit 在两模型间动态选——flash 做高频微调（改进率 100%），pro 偶尔做大改造（改进率 33%）。

**"LLM 缩进丢失这个问题严重吗？具体怎么发现的？"**

非常严重但不明显——所有 candidate 的 score 都是 0，整个 semi run 废了。DeepSeek 生成代码时经常忽略缩进上下文（比如 class 体内应该 4 空格但输出从 0 开始），结果 `def set_limit_order` 变成顶层函数而非类方法，`self.set_limit_order()` 抛 AttributeError。prompt 里看不到完整文件的缩进结构（只看到 EVOLVE-BLOCK 内的片段）。框架的 try/catch 吞了异常返回 None，策略默默不交易。通过对比 best.py 和 baseline.py 的缩进才定位到。修复是在 `replace_mutable_content()` 中自动检测期望缩进 + LLM 输出缩进 + 重新对齐。

**"为什么用子进程而不是直接 import？"**

1. 隔离——LLM 生成的代码可能有语法错误、死循环、恶意代码，子进程挂了不影响主进程
2. 环境干净——不同 candidate 可能有冲突的 import 或全局变量
3. 超时控制——subprocess 超时 600s 自动 kill
4. 论文也是这么设计的——评估器通过命令行调用，写 result.json 通信

### 关键数字
- 2 模型 (flash 75% / pro 25%)
- 2 岛 ring 迁移 10%
- 3 维 MAP-Elites
- Diff 70% / Holistic 30%
- 子进程超时 600s

## Bullet 4: 实验结果

### 面试官可能问

**"-$44K 到 +$5.3K 是在什么条件下？"**

快速验证阶段：2024 年第一周数据（10,080 根 K 线），10 个 LLM 候选，2 并发。Run1 进化 `set_target()`，DeepSeek v4-flash 生成的代码引入了动态 sizing 参数和 momentum 项，把基线从不赚钱变成了小赚。这个结果证明 pipeline 能跑通，但不能和论文的全年 4.81 Sharpe 比。

**"全量实验还没跑完吗？"**

坦白说：进行中。主要瓶颈是全年 52.7 万行回测每次 7 分钟（已经优化了 6x），50 个 candidate 需要约 3 小时。还有工程问题在解决（2GB 内存限制、DeepSeek 限流等），但这不影响框架的正确性——快速和半全量验证已经证明了进化能产生有意义的改进。

**"为什么不用 Sharpe 作为评价指标？"**

论文 3.4 节专门讨论了这个问题：Sharpe 可以被"交易更少但更准"的策略虚高——进化系统会偏向不交易来获取高 Sharpe。而 impact-adjusted PnL 通过冲击惩罚（~k^1.5）既防止无限放大仓位，又保持"真正赚钱"的经济激励。

**"Baseline 为什么是负的？"**

我们的 Forecaster（Ridge 回归 3 个 EMA 特征）预测能力很弱，alpha 信号信噪比低。加上 15bps 手续费和冲击成本，基线策略在 2024 牛市中也亏钱。论文的基线 Sharpe 4.81（全年 +$83K），用了更强的特征和不同的数据源（Polygon 聚合 vs 我们 Binance 单交易所）。这是数据源差异的预期结果，不影响进化方法的有效性验证。

**"你把负 PnL 变成正 PnL，这不就是随便跑都能改进？"**

不是。基线负 PnL 是因为交易太频繁（40 万笔/年），手续费和冲击吃掉了利润。LLM 学会了"少交易、交易大"的策略——从 7845 笔降到 14 笔/周。这不是简单的参数调整，是策略逻辑的结构性改变（LLM 自己加了 signal urgency 判断和 dead band）。

### 关键数字
- 快速: -$44K → +$5.3K（1 周，10 候选）
- 半全量: 改善 99.3%（2 周，30 候选）
- 全量: 进行中

## Bullet 5: 数据

### 面试官可能问

**"数据和论文不一样，怎么说服别人结果有可比性？"**

论文用 Polygon（付费，聚合多交易所，BTCUSD），我用 Binance 公开数据（免费，单交易所，BTCUSDT）。论文自己说了 "do not expect quantitative results will hold out-of-the-box on any real exchange"——这篇论文是方法论验证，不是精确实盘策略。我们验证的也是方法论——"LLM 驱动的进化能否改进交易策略"——这个结论对数据源不敏感。换任何一个有足够流动性的标的、任何一个合理的回测引擎，进化都能找到改进。

**"Binance 公开数据是什么？"**

`data.binance.vision`——Binance 官方提供的历史 K 线数据，每月一个 zip，免费下载无需 API Key。每月约 43K 根 1-min 线，数据质量高（无缺失、无异常），时间覆盖从 2017 年至今。

**"时间切分怎么做的？"**

chronological split，和论文完全一致：train 2022-2023（Ridge 训练），val 2024（进化优化），test 2025-01-01→2025-10-10（OOS 验证，进化循环从未见过）。

### 关键数字
- train: 1,051,120 行
- val: 527,040 行
- test: 407,520 行
- 总: ~2M 行（3.8 年的 1-min K 线）

## 通用追问

**"你从中学到了什么？"**

1. Python 对象的真实代价——一个 dict 300 字节，一个 int 28 字节，差了 10 倍，在大数据量下天壤之别
2. LLM 输出的不可靠性——缩进、语法、截断都需要在管道里加多层防御
3. 进化算法 + LLM 的潜力——不是随机搜索，LLM 能给语义级的改进建议
4. 性能优化要量化——每次改动后跑 benchmark，知道哪里是瓶颈、改动有多大收益

**"这个项目最难的三个问题？"**

1. 冲击模型 O(n²) — 从不可用到 7 分钟，三轮优化
2. LLM 缩进 bug — 所有 candidate score=0，花了几小时才定位到是缩进而非逻辑错误
3. 2GB 内存约束 — 被迫重新设计数据结构，意外发现 npy 连续数组比 dict 小 98%

**"如果重新开始，你会怎么做不一样的？"**

1. 先用 1 天 lookback 而非 30 天——没必要为 <1% 的精度牺牲 30x 性能
2. 先测 LLM 在 10 行代码上的表现——Run2 的 12 行空间太小，早该联合进化
3. 第一天就把 pnl_components 改成 numpy 数组——避免后续所有内存问题
